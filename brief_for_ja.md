⏺ Анализ проекта Bella Vladi Face Protocol

  Что это

  Production-ready Telegram-бот с AI-анализом лица: фото → AI-анализ → персональный визуальный протокол (PNG) → подробный web-отчёт → after-photo. Плюс полноценная админ-панель и CRM.

  Стек

  Backend (apps/backend, ~126 .py файлов)
  - FastAPI 0.115 + Uvicorn
  - aiogram 3.17 (Telegram)
  - SQLAlchemy 2.0 + Alembic (PostgreSQL 16) — 9 миграций
  - Celery 5.4 + Redis 7 — 4 очереди
  - OpenAI, Gemini, Replicate (мульти-провайдер AI)
  - Playwright (HTML → PNG рендер)
  - MediaPipe + OpenCV (face landmarks)
  - boto3 / MinIO для S3-совместимого хранилища

  Frontend (apps/frontend, 29 файлов)
  - React 18 + Vite 6 + TypeScript 5.7
  - TanStack Query, React Router 6, Zustand
  - TailwindCSS 3.4, Recharts, lucide-react

  Infra (docker-compose.yml, 9 сервисов)
  - postgres, redis, minio, backend, frontend
  - 4 раздельных воркера: worker_analysis, worker_report, worker_after_photo, worker_telegram
  - опциональный bot_polling (profile=polling)

  Архитектура (сильные стороны)

  1. Чистое разделение pipeline'а — каждая стадия в своей очереди (анализ → отчёт → after-photo → telegram). Деградация одной не валит остальные.
  2. Слоистая структура backend — ai/, bot/, api/, db/, workers/, reports/, storage/, after_photo/, knowledge/, core/ — модули чётко разграничены.
  3. Decoupling visual layer: analysis_json → protocol_copy_json → normalize() → template.html → PNG. Сырой AI-output не попадает напрямую в визуал — есть нормализация под layout.
  4. Mock mode (AI_FORCE_MOCK=true) + mock load-test на 500 заявок без расходов на API.
  5. Storage abstraction — local/S3 переключается флагом STORAGE_DRIVER.
  6. Хорошая операционная зрелость: healthcheck, Redis-lock на setWebhook от race condition нескольких uvicorn-воркеров, progress-сообщения в Telegram, retry/QC для after-photo.
  7. Knowledge base + Prompt templates в БД — менять AI-логику можно без деплоя.
  8. Подробный README (~1000 строк, как playbook): production checklist, scaling, load-test, env vars.

  ❌ Cloud Run тебе не подходит (в чистом виде)

  Твоя интуиция «чтобы сервер не упал» правильная, но Cloud Run — неправильный инструмент именно для этого проекта. Вот почему:

  1. Celery-воркеры — это постоянно работающие процессы, а не «запрос-ответ». Cloud Run скейлится в ноль и убивает инстанс без HTTP-трафика. Воркеры просто умрут. Это его фундаментальная несовместимость с очередями.
  2. Образ backend ~2 ГБ (mediapipe + jax + opencv + playwright + chromium). Cloud Run cold start на таком образе — это 5-15 секунд. Telegram-вебхук за это время словит таймаут.
  3. Чтобы убрать cold start, ставишь min-instances=1 → платишь круглосуточно → теряешь весь смысл Cloud Run (scale-to-zero).
  4. Больше движущихся частей = больше шансов что-то сломается у соло-оператора.

  Cloud Run хорош для: stateless, лёгких, «бурстовых» HTTP-API. У тебя — stateful асинхронный pipeline с тяжёлым ML. Это другое.

  Реальное преимущество S3 — общий доступ с нескольких машин. А ты на одной VM, и при 5-10k лидов воронки тебе не понадобится вторая. Значит ты платил бы (усилиями на рефакторинг) за фичу, которой не воспользуешься.

  Когда S3/GCS реально понадобится

  Только если в будущем выйдешь на горизонтальный масштаб — несколько backend/воркер-машин, которым нужен общий доступ к файлам. Вот тогда стоит делать Путь 1 (полный рефакторинг). Но это сценарий «вырос в 10-50 раз», не старт.




  Что такое Persistent Disk в Google Cloud

  Аналогия: представь, что VM (виртуальная машина) — это компьютер. Persistent Disk — это внешний жёсткий диск, который ты к нему подключаешь.

  Ключевая идея — диск живёт отдельно от компьютера:

  ┌─────────────────┐         ┌──────────────────────┐
  │   VM (bella-app) │ ◄─────► │  Persistent Disk      │
  │   = "компьютер"  │ подключ.│  (bella-data, 50 ГБ)  │
  │                  │         │  = "внешний HDD"      │
  │  можно удалить,  │         │  данные остаются      │
  │  пересоздать     │         │  даже если VM умерла  │
  └─────────────────┘         └──────────────────────┘

  Почему это важно:

  ┌─────────────────────────────────────────┬─────────────────────┬────────────────────────────────────────────┐
  │                                         │ Загрузочный диск VM │        Persistent Disk (отдельный)         │
  ├─────────────────────────────────────────┼─────────────────────┼────────────────────────────────────────────┤
  │ Удалил/пересоздал VM                    │ ❌ данные пропали   │ ✅ данные остались                         │
  ├─────────────────────────────────────────┼─────────────────────┼────────────────────────────────────────────┤
  │ Надёжность                              │ обычная             │ Google реплицирует его в нескольких местах │
  ├─────────────────────────────────────────┼─────────────────────┼────────────────────────────────────────────┤
  │ Снапшоты (бэкап)                        │ можно               │ ✅ можно, легко                            │
  ├─────────────────────────────────────────┼─────────────────────┼────────────────────────────────────────────┤
  │ Можно отцепить и подключить к другой VM │ сложно              │ ✅ да, как USB переткнуть                  │
  └─────────────────────────────────────────┴─────────────────────┴────────────────────────────────────────────┘

  То есть это долговечное хранилище, которое не зависит от судьбы конкретной VM. Сломалась VM ночью, ты создал новую за 2 минуты, подключил тот же диск — все фотки и база на месте.

  Два типа дисков в GCP — не перепутай

  Google даёт два разных вида хранилища, и важно понимать разницу:

  1. Persistent Disk (то, что мы используем) — это блочное устройство = обычная файловая система. Видится в системе как /mnt/data, в нём папки и файлы, как на флешке. Pipeline пишет туда abs_path() → реальные файлы. ✅ Подходит.
  2. Cloud Storage (GCS) — это объектное хранилище, S3-подобное. Там нет файловых путей, только «ключ → объект» через API. Pipeline так не умеет (мы это обсудили). ❌ Не подходит без рефакторинга.

  Persistent Disk = «жёсткий диск». GCS = «облачное API для файлов». Нам нужен первый.

  Как это работает в твоём проекте

  Теперь конкретика. Вот цепочка от создания до работы:

  Шаг 1 — создаёшь и подключаешь диск (один раз)

  gcloud compute disks create bella-data --size=50GB --zone=europe-west1-b
  gcloud compute instances attach-disk bella-app --disk=bella-data --zone=...
  gcloud compute disks create bella-data --size=50GB --zone=europe-west1-b
  gcloud compute instances attach-disk bella-app --disk=bella-data --zone=...

  gcloud compute disks create bella-data --size=50GB --zone=europe-west1-b
  gcloud compute instances attach-disk bella-app --disk=bella-data --zone=...
  Теперь к VM подключён «внешний HDD» на 50 ГБ.

  Шаг 2 — монтируешь его в /mnt/data (один раз)

  sudo mkfs.ext4 ... /dev/.../google-bella-data    # отформатировать (ТОЛЬКО на новом!)
  sudo mount /dev/.../google-bella-data /mnt/data   # подключить в папку
  После этого всё, что ты кладёшь в /mnt/data на VM — физически пишется на этот отдельный диск. Создаёшь две папки:
  sudo mkdir -p /mnt/data/postgres   # сюда ляжет база
  sudo mkdir -p /mnt/data/storage    # сюда лягут фото/PNG

  Шаг 3 — docker подключает эти папки в контейнеры

  Это уже сделано в docker-compose.prod.yml, который я написал. Он говорит:

  /mnt/data/storage   (на диске VM)  ──►  /app/storage   (внутри backend и воркеров)
  /mnt/data/postgres  (на диске VM)  ──►  /var/lib/postgresql/data (внутри postgres)

  Это называется bind-mount: папка с диска «прокидывается» внутрь контейнера. Контейнер думает, что пишет в /app/storage, а на самом деле байты летят на персистентный диск.

  Что происходит при реальной работе бота

  1. Юзер шлёт фото в Telegram
          ↓
  2. backend: local_storage.save_bytes("photos/123.jpg", data)
     → пишет в /app/storage/photos/123.jpg внутри контейнера
     → physically → /mnt/data/storage/photos/123.jpg на персистентном диске  ✅
          ↓
  3. worker_analysis: local_storage.abs_path("photos/123.jpg")
     → "/app/storage/photos/123.jpg"  ← реальный путь, opencv/mediapipe открывают файл
          ↓
  4. Playwright рендерит PNG → тоже в /app/storage → тоже на диск
          ↓
  5. Postgres пишет всю базу лидов → /var/lib/postgresql/data → тоже на этот диск

  Всё — и файлы, и база — оседает на одном персистентном диске /mnt/data.

  Что это даёт на практике

                    ┌────────────────────────────────┐
     VM сломалась   │  Persistent Disk bella-data    │
     / пересоздал ──┤  /mnt/data/storage  (фото,PNG) │ ← ВСЁ ЦЕЛО
                    │  /mnt/data/postgres (база)     │
                    └────────────────────────────────┘
                             │
                             ▼
                    снапшо[118;1:3uт раз в день = бэкап,
                    можно откатиться на любой день

  - Пересоздал VM → подключил тот же диск → всё на месте.
  - Снапшот диска (gcloud compute disks snapshot) = полный бэкап фоток + базы одной командой.
  - Случайно удалил данные → восстановил из вчерашнего снапшота.

  Коротко

  - Persistent Disk = внешний жёсткий диск Google, живёт отдельно от VM, надёжный, снапшотится.
  - Это файловая система (/mnt/data), а не облачное API — поэтому твой pipeline работает с ней без единой правки кода.
  - Монтируешь его в /mnt/data, docker прокидывает папки внутрь контейнеров, и все фото/PNG/база физически пишутся на него.
  - Снапшоты раз в день = бэкап. VM можно убивать и пересоздавать — данные переживут.
